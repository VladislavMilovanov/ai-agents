# Техническое видение проекта

## 1. Технологии и инструменты

Проект реализуется на Python 3.12 как Telegram-бот на aiogram (режим polling).
Управление зависимостями и запуском выполняется через uv.
Для работы с LLM используется официальный openai client с подключением к OpenRouter как провайдеру моделей.
Локальная разработка и запуск поддерживаются в двух режимах: нативно через make + uv и контейнерно через Docker/Compose + make.
В качестве минимальных инструментов качества используются ruff и pytest.

## 2. Принцип разработки

Разработка ведется по принципу KISS: реализуется только функциональность, необходимая для проверки идеи Telegram LLM-ассистента в роли технического консультанта по состоянию оборудования.
Код строится в простом OOP-подходе со строгим правилом «1 класс = 1 файл».
Преждевременные абстракции и усложняющие архитектурные решения не допускаются: новые уровни, паттерны и сущности добавляются только при реальной необходимости.
Реализация идет малыми итерациями: сначала рабочий end-to-end сценарий, затем локальные улучшения без разрушения простоты решения.

## 3. Структура проекта

Структура проекта организована вокруг минимальных прикладных модулей: конфигурация, Telegram-слой, LLM-интеграция, сервисная логика, модели данных и логгирование.
Исходный код размещается в `src/`, тесты — в `tests/`, документация — в `docs/`.
В корне проекта находятся файлы управления зависимостями и запуском (`pyproject.toml`, `uv.lock`), сценарии разработки (`Makefile`) и контейнеризации (`Dockerfile`, `docker-compose.yml`), а также шаблон переменных окружения (`.env.example`).
Внутри кода соблюдается правило «1 класс = 1 файл».

## 4. Архитектура проекта

Архитектура проекта строится как простой слоеный поток без лишних абстракций: Telegram-обработчик, сервисная оркестрация, LLM-клиент и инфраструктурные модули конфигурации/логгирования.
Telegram-слой отвечает за прием и отправку сообщений, сервисный слой — за прикладной сценарий обработки, LLM-слой — за вызов модели через OpenRouter.
Конфигурация и логгирование вынесены в отдельные инфраструктурные модули и используются всеми слоями.
Основной поток обработки: входящее сообщение Telegram -> handler -> chat service -> LLM client -> chat service -> ответ в Telegram.

## 5. Модель данных

Для MVP используется минимальная модель данных: входящее сообщение пользователя, ответ ассистента и объединяющая пара сообщений (chat turn).
На первом этапе постоянное хранилище данных не используется: решение работает без базы данных для снижения сложности.
Контекст диалога при необходимости поддерживается в памяти процесса как ограниченный буфер последних сообщений пользователя.
Модель данных остается плоской и прикладной, без избыточных сущностей и инфраструктурных слоев.

## 6. Работа с LLM

Интеграция с LLM реализуется через официальный openai client с подключением к OpenRouter по `base_url` и API-ключу из переменных окружения.
Для MVP используется одна настраиваемая модель, задаваемая в конфиге приложения, без динамического выбора провайдеров и моделей.
Запрос к модели формируется из системного промпта (роль технического консультанта по состоянию оборудования), текущего сообщения пользователя и, при необходимости, краткого контекста из памяти процесса.
Системный промпт задаёт роль эксперта: анализ данных за период, оценка режимов работы, диагностика отклонений и аномалий, прогноз состояния оборудования.
Параметры генерации задаются в конфигурации и остаются стабильными на первом этапе.
При сбоях обращения к LLM пользователь получает безопасный технический fallback-ответ, а детали ошибки фиксируются в логах.

## 7. Сценарии работы

В MVP поддерживаются базовые пользовательские сценарии в контексте роли технического консультанта по оборудованию.

Основные прикладные сценарии:
- пользователь описывает параметры работы оборудования за период — бот анализирует и резюмирует состояние;
- пользователь задаёт вопрос о причинах отклонения или неисправности — бот формулирует диагностику;
- пользователь запрашивает прогноз — бот оценивает дальнейшее состояние на основе предоставленных данных.

Диалог поддерживается через историю сообщений: пользователь может уточнять параметры, добавлять данные по периоду, задавать уточняющие вопросы в рамках одной сессии.

Защитные сценарии: обработка временной недоступности LLM и обработка неподдерживаемого/пустого ввода без остановки процесса бота.
Сценарии ограничены необходимым минимумом для проверки идеи и не включают сложные ветвления диалога.

## 8. Подход к конфигурированию

Конфигурирование строится на переменных окружения с локальной загрузкой из `.env` для разработки.
Все параметры приложения агрегируются в единый объект `Settings`, который инициализируется при старте и передается в рабочие компоненты.
На этапе MVP не вводятся многослойные схемы конфигов и отдельные профили окружений; используется единый, прозрачный набор параметров.
Обязательные параметры (токены, модель, базовые LLM и logging-настройки) валидируются при запуске, чтобы исключить скрытые ошибки во время выполнения.

## 9. Подход к логгированию

Для MVP используется стандартный модуль `logging` с единой точкой настройки при запуске приложения.
Логи выводятся в стандартный поток `stdout` в простом структурированном формате (время, уровень, модуль, сообщение).
Уровень детализации задается через переменную `LOG_LEVEL`.
Логируются ключевые этапы работы бота и ошибки интеграций, при этом секреты и чувствительные значения не попадают в лог.

## 10. Подход к сборке и деплою

Сборка и запуск проекта поддерживаются в двух простых вариантах: нативный локальный режим через `uv` и `make`, а также контейнерный режим через `Docker`/`docker compose` и `make`.
`Makefile` используется как единая точка входа для основных операций разработки: установка зависимостей, запуск, lint, тесты, сборка и управление контейнером.
Контейнеризация реализуется одним сервисом бота без дополнительной оркестрации.
Переменные окружения передаются через `.env`, что обеспечивает одинаковый подход к запуску в обоих режимах.
