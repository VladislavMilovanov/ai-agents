# Техническое видение проекта

## 1. Технологии и инструменты

Проект реализуется на **Python 3.12** как Telegram-бот на **aiogram** (асинхронный API, режим **polling**).
Управление зависимостями и запуском — **uv**; сценарии разработки — **Makefile**.
Для LLM и RAG используется **LangChain** и официальный **openai**-клиент с `base_url`, указывающим на **OpenRouter** (`https://openrouter.ai/api/v1`), и API-ключом из окружения.
Модели чата и (при необходимости) эмбеддингов задаются в **`.env`** (имена и параметры — в едином `Settings`).
Локальный запуск: **make** + **uv**; опционально контейнер (**Docker** / **Compose**) без усложнения контрактов.
Качество кода: **ruff**, **pytest**.

Референс реализации цепочки RAG с **query transformation**: ноутбук **`naive-rag.ipynb`**, итоговая сборка **`rag_query_transform_chain`** — ориентир по шагам пайплайна, не как обязательная копия «строка в строку».

## 2. Принцип разработки

**KISS** и **YAGNI**: только функциональность, нужная для сквозного сценария «индексация → диалог в Telegram → ответ по RAG»; без лишних слоёв и абстракций «на вырост».
**Один класс — один файл**, простой OOP.
Сначала рабочий **end-to-end**, затем точечные улучшения.
Преждевременные БД, очереди, микросервисы и универсальные «движки» не вводятся.

## 3. Структура проекта

Исходный код — **`src/`**, тесты — **`tests/`**, документация — **`docs/`**.
Корпус для индексации — **`data/`**:
- `ouk_potrebitelskiy_kredit_lph.pdf`
- `usl_r_vkladov.pdf`
- `sberbank_help_documents.json`

В корне: `pyproject.toml`, `uv.lock`, `Makefile`, шаблон **`.env.example`**, при необходимости Docker-файлы.
Экспериментальный ноутбук с референсом пайплайна может лежать в репозитории (например `naive-rag.ipynb`); продакшн-поток бота реализуется в `src/`, а не в ноутбуке.

## 4. Архитектура и поток данных

Слои без избыточной вложенности:

- **Telegram**: приём/отправка сообщений, команды.
- **Сервис диалога / RAG**: оркестрация истории, вызов RAG-цепочки, формирование ответа пользователю.
- **Индексация**: загрузка файлов из `data/`, разбиение на чанки, эмбеддинги, запись во **векторное хранилище в памяти процесса**.
- **Инфраструктура**: `Settings`, логирование, инициализация клиентов OpenRouter.

Основной поток вопроса пользователя: **handler → сервис → (query transformation → retriever → LLM с контекстом) → ответ в Telegram**.
Команды **`/index`** и **`/index_status`** обрабатываются в том же боте и вызывают логику переиндексации / отчёта о состоянии индекса.

## 5. Данные и хранилища

**Постоянная БД не используется** (MVP).
**Векторное хранилище:** **`InMemoryVectorStore`** (LangChain) — жизнь индекса = жизнь процесса; после рестарта нужна повторная индексация (в т.ч. автоматическая при старте).
Диалоговая **история** хранится в памяти процесса ограниченным буфером per-user (как сейчас), но **в форме сообщений LangChain**: `HumanMessage` / `AIMessage` / при необходимости `SystemMessage` — чтобы тот же формат использовался в query transformation и в финальном запросе к модели.

## 6. Индексация и команды бота

- **`/index`**: полная переиндексация корпуса из `data/` (пересборка векторного хранилища).
- **При старте бота**: выполняется **полная переиндексация** (как при `/index`), чтобы к моменту приёма сообщений индекс был готов.
- **`/index_status`**: краткий статус — как минимум **число чанков** в текущем индексе (и при ошибке/пустом индексе — понятное сообщение).

Детали сплиттера и размера чанка — разумный дефолт + при необходимости параметры в конфиге; без оверинжиниринга.

## 7. RAG и работа с LLM

Перед поиском выполняется **query transformation** с учётом **истории диалога** (как в референсной цепочке `rag_query_transform_chain`).
Из ретривера извлекается **топ-K** документов; **K** задаётся переменной окружения **`RETRIEVER_K`** (валидируется при старте вместе с остальными обязательными параметрами).
Финальный ответ LLM строится по **истории** (LangChain messages) и **контексту** из отобранных чанков; системное сообщение задаёт правила (опора на источники, честность при отсутствии фактов и т.д. — по смыслу MVP).

Один настраиваемый путь к API (OpenRouter); отдельные идентификаторы моделей для чата и эмбеддингов — через `Settings`, если используются разные модели.

При сбоях вызова модели или индексации пользователь получает **безопасный** короткий ответ; подробности — только в **логах** (без секретов).

## 8. Сценарии работы

- Пользователь задаёт вопрос по тематике документов — бот отвечает с опорой на релевантные чанки.
- Пользователь уточняет вопрос в том же чате — учитываются история и query transformation.
- Администратор/пользователь запускает **`/index`** для обновления индекса после смены файлов в `data/`.
- **`/index_status`** показывает, что индекс построен (число чанков).
- Ошибки API и пустой/бессмысленный ввод обрабатываются без падения процесса.

## 9. Конфигурирование

Единый объект **`Settings`**: переменные окружения + опционально `.env` для локальной разработки.
Обязательные параметры (токен бота, ключ API, `OPENAI_BASE_URL` / OpenRouter, модели, **`RETRIEVER_K`**, лимиты истории, уровень логов и т.д., релевантные MVP) валидируются **при старте**.
Без многоуровневых профилей конфигов.

## 10. Логирование

Стандартный **`logging`**, вывод в **`stdout`**, уровень из конфига.
Логируются этапы индексации, вызовы цепочки (без утечки секретов и персональных данных в избытке).

## 11. Сборка и запуск

**make** + **uv** как основной путь; Docker — опционально, без дублирования и рассинхрона с единым `.env`.
Команды установки зависимостей, запуска, линтера и тестов остаются простыми и документированными в README.
