### Итерация 1 — Бот в Telegram

### Итерация 2 — Ответ через LLM

Цель: на текстовый вопрос приходит ответ, сформированный моделью.

- [x] Связка «сообщение пользователя → запрос к модели → ответ в чат»

### Итерация 3 — Роль ассистента

Цель: ответы соответствуют заданной роли из системного промпта.

- [x] Роль задаётся в настройках и участвует в каждом запросе к модели
- [x] Смена роли в конфиге заметно меняет стиль или содержание ответов

**Как проверить:** два разных описания роли — два разных «характера» ответов на один и тот же вопрос.

### Итерация 4 — Ошибки и пустой ввод

Цель: бот не падает; пользователь получает понятную реакцию при сбоях и мусорном вводе.

- [x] Сбой обращения к модели: безопасное сообщение пользователю, детали только в логах
- [x] Пустой или заведомо неподходящий ввод: корректная реакция без обрыва диалога

**Как проверить:** симулировать недоступность API и отправить пустое/бессмысленное сообщение — бот отвечает стабильно.

### Итерация 5 — Память диалога

Цель: ответы учитывают недавние реплики в рамках общения с пользователем.

- [x] В запрос к модели попадает ограниченная история последних сообщений
- [x] Лимит истории задан разумно (не раздувает контекст бесконечно)

**Как проверить:** ссылка на предыдущую фразу в следующем сообщении — модель «помнит» контекст.

**Как проверить:** чистое окружение → по инструкции запуск → тесты и линтер проходят.

## Спринт 2 — итерации (активные чеклисты)

### Итерация 8 — Индексация и статус

Цель: корпус из `data/` стабильно попадает в `InMemoryVectorStore`; пользователь видит команды переиндексации и статистику.

- [x] Загрузка `ouk_potrebitelskiy_kredit_lph.pdf`, `usl_r_vkladov.pdf`, `sberbank_help_documents.json`
- [x] Разбиение на чанки, эмбеддинги через OpenRouter/openai client, запись в `InMemoryVectorStore`
- [x] Полная переиндексация при **старте** процесса бота
- [x] Команда **`/index`** — полная переиндексация по запросу
- [x] Команда **`/index_status`** — минимум **число чанков** в индексе

**Как проверить:** после старта и после `/index` — `/index_status` показывает ожидаемое ненулевое число чанков; логи без падений.

### Итерация 9 — RAG-цепочка

Цель: поведение как у референса `rag_query_transform_chain`: преобразование запроса с учётом истории, топ-K, ответ по контексту.

- [x] **Query transformation** перед retrieval (сигнал из истории диалога)
- [x] Retriever: **топ-K**, K из **`RETRIEVER_K`** (`.env` / `Settings`)
- [x] LLM формирует ответ по извлечённому контексту + правилам системного промпта
- [x] Структура цепочки согласована с **`naive-rag.ipynb`** (те же этапы, допустимы адаптации под async/бота)

**Как проверить:** на фиксированном корпусе вопрос с ответом из документа — в ответе явная опора на формулировки из чанка (smoke вручную или узкий тест с моком ретривера).

### Итерация 10 — Диалог и интеграция в бота

Цель: обычные сообщения в чате идут через RAG; история — в формате LangChain messages.

- [x] История хранится/передаётся как **`HumanMessage` / `AIMessage`** (и при необходимости **`SystemMessage`**)
- [x] Сервис/handler связывает Telegram, историю и RAG-цепочку без дублирования логики
- [x] `/index` и `/index_status` не ломают состояние диалога

**Как проверить:** уточняющий вопрос после первого — query transformation меняет поиск осмысленно; ответы не обрываются.

### Итерация 11 — Конфиг, тесты, документация

Цель: запуск и сопровождение понятны без чтения кода.

- [x] **`.env.example`**: OpenRouter base URL, модели, **`RETRIEVER_K`**, лимиты истории, прочее обязательное для RAG
- [x] Валидация **`Settings`** при старте на полноту параметров RAG
- [x] Тесты: критический путь (индекс из фикстур/моков, вызов цепочки при необходимости с моком LLM)
- [x] **README**: как положить файлы в `data/`, как запустить, команды `/index` и `/index_status`

**Как проверить:** чистое venv/`uv sync` → по README запуск → ручная проверка трёх команд и диалога.
