# Техническое видение проекта

## 1. Технологии и инструменты

Проект реализуется на Python 3.12 как Telegram-бот на aiogram (режим polling).
Управление зависимостями и запуском выполняется через uv.
Для работы с LLM/VLM используется официальный openai client с возможностью выбора провайдера через `base_url` (OpenRouter или Ollama).
Локальная разработка и запуск поддерживаются в двух режимах: нативно через make + uv и контейнерно через Docker/Compose + make.
В качестве минимальных инструментов качества используются ruff и pytest.

## 2. Принцип разработки

Разработка ведется по принципам KISS и YAGNI: реализуется только функциональность, необходимая для проверки идеи персонального финансового советника.
Код строится в простом OOP-подходе со строгим правилом «1 класс = 1 файл».
Преждевременные абстракции и усложняющие архитектурные решения не допускаются.
Реализация идет малыми итерациями: сначала рабочий end-to-end сценарий учета транзакций, затем локальные улучшения.

## 3. Структура проекта

Структура проекта организована вокруг минимальных прикладных модулей: конфигурация, Telegram-слой, LLM/VLM-интеграция, сервисная логика, модель транзакций, in-memory хранилище и логгирование.
Исходный код размещается в `src/`, тесты — в `tests/`, документация — в `docs/`.
В корне проекта находятся `pyproject.toml`, `uv.lock`, `Makefile`, `Dockerfile`, `docker-compose.yml` и `.env.example`.
Внутри кода соблюдается правило «1 класс = 1 файл».

## 4. Архитектура проекта

Архитектура строится как простой поток: Telegram handler -> сервис -> LLM/VLM-клиент -> in-memory журнал транзакций -> ответ в Telegram.
Telegram-слой отвечает за прием текстов и фотографий чеков.
Сервисный слой маршрутизирует сценарий: извлечение транзакций, запрос отчета, обработка ошибок.
LLM/VLM-слой отвечает за structured output и multimodal-анализ изображения.
Конфигурация и логгирование вынесены в отдельные инфраструктурные модули.

## 5. Модель данных

Для MVP используется минимальная модель транзакции:
- дата и время операции;
- направление (`income`/`expense`);
- сумма;
- тип траты (`daily`/`periodic`/`one_time`);
- категория (например, продукты, рестораны, такси, образование, путешествия);
- описание.

Постоянное хранилище на этапе MVP не используется.
Журнал транзакций хранится in-memory с привязкой к пользователю Telegram.

## 6. Работа с LLM и VLM

Интеграция с моделями реализуется через openai client и конфигурируемые `api_key` + `base_url`.
Текстовые сообщения обрабатываются LLM с structured output: результатом является валидированная структура транзакции или команда на отчет.
Изображения чеков обрабатываются VLM напрямую из фото (без ручного OCR-пайплайна), также с structured output.
Модель(и) задаются в конфиге и могут быть переключены между OpenRouter и локальным Ollama.
Для устойчивости запросов используются минимальные таймауты и ограниченные повторы.
При сбоях пользователь получает безопасный fallback-ответ, детали ошибки фиксируются только в логах.

## 7. Сценарии работы

В MVP поддерживаются базовые прикладные сценарии:
- пользователь отправляет текст о доходе/расходе — бот извлекает структуру и записывает транзакцию;
- пользователь отправляет фото чека — бот извлекает данные и записывает транзакцию;
- пользователь запрашивает отчет — бот возвращает баланс, разрез по периоду и категории;
- пользователь уточняет данные — бот учитывает контекст последнего диалога.

Защитные сценарии: недоступность модели, пустой/неподходящий ввод, невозможность извлечь корректные данные.

## 8. Подход к конфигурированию

Конфигурация строится на переменных окружения с локальной загрузкой из `.env`.
Все параметры агрегируются в единый объект `Settings`.
Обязательные параметры валидируются при запуске.
В конфиге отдельно задаются:
- провайдер/`base_url`;
- API-ключ;
- текстовая модель;
- визуальная модель;
- системный промпт финансового ассистента;
- таймаут и число повторов запросов к моделям;
- лимит размера изображения чека;
- лимиты истории и логгирование.

## 9. Подход к логгированию

Используется стандартный `logging` с единой инициализацией в точке входа.
Логи пишутся в `stdout` в формате: время, уровень, модуль, сообщение.
Секреты и чувствительные финансовые данные не логируются.
Логируются ключевые этапы: входящее сообщение, результат извлечения (без персональных деталей), ошибки интеграции.

## 10. Подход к сборке и деплою

Сборка и запуск поддерживаются через `uv` + `make`, а также через Docker/Compose + make.
`Makefile` остается единой точкой входа для `install`, `run`, `lint`, `test`, контейнерных команд.
Контейнеризация реализуется одним сервисом бота без дополнительной оркестрации.
Переменные окружения передаются через `.env` в обоих режимах запуска.
